# Translation Pipeline v1

Manga CMS should support machine-assisted translation without treating machine
output as canonical manga content. This document defines a staged translation
pipeline that uses the existing Translation Pack Draft import flow as the
ingestion point for machine, LLM, and human translation rows.

The default unit of machine translation in this spec is one Page. Page-level
translation gives the model enough local context to preserve dialogue flow
while keeping prompt size, cost, and validation errors bounded. Episode-level
translation can be revisited later as a batching optimization. Bubble-level
translation without Page context is not the default because it loses speaker,
Panel, and neighboring Bubble context.

## Existing Pipeline Fit

The current repository already has most of the required pipeline:

1. `POST /admin/pack-drafts/{packDraftId}/translation-import` accepts
   normalized JSON/CSV translation rows.
2. `buildTranslationPackDraftImportPlan` validates rows against canonical
   `Episode.pages[].bubbles[]` by `bubble_id`.
3. Import planning reports `unmatched_bubble`, `duplicate_bubble`,
   `missing_bubble`, `existing_entry_conflict`, and `source_text_mismatch`.
4. Applied imports write runtime `TRANSLATION` Pack Draft entries only.
5. Pack Draft review status remains separate from canonical Pack publication.
6. `POST /admin/pack-drafts/{packDraftId}/export` writes a canonical
   `packs/{packId}/pack.json` manifest.
7. Public Reader payloads expose only canonical Pack manifests with
   `isPublished: true` through `availablePacks`.

This means machine translation does not need a new public Reader route or a new
canonical content path. The machine output can be converted into the same
translation-import rows used by human spreadsheet imports.

## Non-Goals

- Do not write machine output to `contents/` or `Bubble.textOriginal`.
- Do not create canonical Pack manifests directly from machine output.
- Do not expose runtime Pack Draft text in public Reader payloads.
- Do not let public Reader requests generate or save translations.
- Do not add a payment, tenant, hosted creator, or commercial platform feature.
- Do not make LLM/VLM providers required for self-hosted operation.

## Translation Origin Metadata

The minimal backward-compatible addition is entry-level metadata. Pack Draft
entries and Pack manifest entries already allow `metadata?: Record<string,
unknown>`, and published Reader Pack entries omit `metadata`. Therefore origin
metadata can be carried through review and canonical Pack export without leaking
to Reader payloads.

Recommended metadata shape:

```json
{
  "translation_origin": "machine",
  "provider": "gemini",
  "model": "gemini-2.0-flash",
  "confidence": 0.82,
  "generated_at": "2026-06-13T00:00:00.000Z",
  "source": "translation_import",
  "source_format": "json"
}
```

Origin values:

- `machine`: generated by a machine or LLM translation provider.
- `human`: edited or authored by a human reviewer or translator.
- `imported`: imported without a stronger known origin. This is the default
  interpretation for existing rows that lack `translation_origin`.

`translation_origin` should live under `metadata` for v1. A dedicated top-level
field can be considered later only if multiple workflows need strongly typed
queries across Pack Drafts and canonical Pack manifests.

## Page-Level LLM Translation Shape

The translation runner should build one request per Page by default:

1. Read the target Episode from canonical content.
2. Sort Panels and Bubbles using the canonical reading order and the
   [reading-order-spec.md](reading-order-spec.md) rules where inferred ordering
   is needed.
3. Build a Page script with stable `bubbleId` markers, speaker labels when
   available, `bubbleType`, Panel boundaries, and source `textOriginal`.
4. Add glossary and character voice notes if configured.
5. Ask the provider for structured output keyed by `bubbleId`.
6. Validate that output Bubble IDs exactly match the input set for that Page.
7. Convert matched rows into translation-import JSON entries with
   `translation_origin: "machine"`.

The model output should be treated as a candidate. Missing, extra, duplicate,
or malformed Bubble IDs must not be auto-corrected into canonical data. They
should be surfaced as import warnings or blocking issues for CMS review.

## Provider Boundary

The existing `packages/ingestion` provider abstraction is useful precedent, but
translation should not be forced into the same interface immediately.

Recommendation:

- Share provider conventions: provider ID, model ID, prompt version,
  diagnostics, generated timestamp, noop fallback, and env-based secrets.
- Keep translation-specific input/output types separate from region detection
  and OCR alignment types.
- Reuse `packages/ingestion` only if it remains a provider/workflow package
  rather than a manga-image-ingestion-only package.

Provider configuration is optional. If no provider is configured, the
translation runner should be a noop and return an actionable diagnostic. The
public Reader must never call a provider.

## CMS Review Flow

Machine-origin entries should be visible as candidates, not as final
translations:

1. Import or generate rows into a runtime `TRANSLATION` Pack Draft.
2. Show machine-origin rows as "machine translated / needs review" in CMS.
3. Continue to show existing fit guidance and `source_text_mismatch` warnings.
4. When a human edits an entry, update metadata to `translation_origin:
   "human"` and preserve provider metadata as historical provenance if useful.
5. Use existing Pack Draft statuses for review and approval.
6. Export only approved/published Pack Drafts to canonical `packs/` manifests.

The first implementation can keep "origin becomes human on edit" as a CMS rule
without changing Reader behavior.

## Boundary Checks

- Canonical `contents/` remains the source of truth for source manga structure.
- `Bubble.textOriginal` remains source text, not translated text.
- Runtime Pack Drafts are review state, not public content.
- Canonical Pack export remains the explicit publication boundary.
- Public Reader `availablePacks` remains limited to `isPublished: true`
  canonical Pack manifests.
- Pack metadata such as `translation_origin`, provider, model, confidence, and
  prompt version must not be included in public Reader payloads unless a future
  public contract explicitly opts into it.
- Provider keys and prompts are deployment configuration, not repository data.

## Open Questions And Recommendations

1. **Origin field location**: use `metadata.translation_origin` for v1. It is
   backward-compatible and does not affect Reader payloads.
2. **Translation granularity**: use Page as the default. Allow a small previous
   Page context window later if needed, but do not make Episode-wide prompts
   the default.
3. **Glossary and translation memory storage**: start with runtime file-backed
   review state or DB-backed runtime state. Do not store project-specific
   glossary secrets or private editorial notes in canonical Pack manifests by
   default.
4. **Provider abstraction**: share conventions with ingestion providers, but
   define translation-specific types.
5. **Cost and rate control**: place controls on admin-triggered batch jobs, not
   public Reader requests. Batches should be explicit, cancellable, and scoped
   to Series/Episode/language.

## Phase Plan

### Phase 1: Origin Metadata

- Extend translation-import entry input to accept optional
  `translation_origin`, `provider`, `model`, `confidence`, and `generated_at`.
- Store those values under Pack Draft entry `metadata`.
- Preserve existing imports by treating missing origin as `imported`.
- Surface origin in CMS import preview and Translation Workspace.
- Update schemas, OpenAPI, and API contract together.

No Reader behavior changes in this phase.

### Phase 2: Page Script Builder And Provider Output Validation

- Build Page scripts from canonical Bubble order.
- Add translation-specific provider input/output types.
- Add noop provider behavior for unconfigured deployments.
- Validate output Bubble IDs for missing, extra, duplicate, and malformed IDs.
- Convert valid output to translation-import JSON rows.

No Pack Draft writes are required in the first implementation of this phase.

### Phase 3: Admin Batch Runner To Pack Draft Import

- Add an admin-only runner that creates machine-origin import rows and applies
  them through the existing translation-import route or service.
- Keep public Reader unreachable from this runner.
- Reuse existing fit guidance and import mismatch checks.
- Require explicit Page selection and cap each request to a bounded number of
  Pages.
- Treat provider output as page-level all-or-nothing: a Page with missing,
  extra, duplicate, or malformed Bubble IDs is skipped, while other valid Pages
  in the same request may still become Pack Draft import rows.
- If no provider is configured, return actionable diagnostics and do not mutate
  the Pack Draft.

### Phase 4: CMS Human Review

- Mark machine-origin rows as unreviewed.
- When a reviewer edits a row, set origin to `human`.
- Keep Pack Draft approval and export flow unchanged.

### Phase 5: Reader Experiments As Design Only

Unreviewed machine translations should not appear in Reader by default. Any
future "machine translation preview" should be opt-in, clearly labeled, and
separate from published official Pack behavior.
